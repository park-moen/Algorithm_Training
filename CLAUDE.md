```

```

## 풀이 검토 방법

사용자가 풀이 파일을 제출하면 아래 순서로 검토합니다.

### 1단계: 정확성

- 문제의 모든 예시 케이스를 통과하는지 확인
- 엣지 케이스 누락 여부 확인 (빈 배열, 길이 1, 음수, 중복 등)
- **동작은 하지만 의도와 다른 코드**가 있는지 확인 (우연히 통과하는 케이스)

### 2단계: 시간/공간 복잡도

- 현재 풀이의 Big-O 분석
- 더 나은 복잡도가 가능한지 검토
- 불필요한 자료구조 변환(예: `[...s]`) 여부 확인

### 3단계: 코드 품질

아래 패턴을 중점적으로 확인합니다.

#### 자주 나오는 안티패턴

| 패턴                             | 문제                 | 개선                    |
| -------------------------------- | -------------------- | ----------------------- |
| `[...s].forEach(...)`            | 불필요한 배열 생성   | `for...of s`            |
| `forEach` 안의 `return false`    | 함수를 종료하지 않음 | `for...of` + `return`   |
| 값을 쓰지 않는 `Map`             | 의미 불명확          | `Set` 사용              |
| `map.entries()` 순회 + `indexOf` | O(n²) 가능성         | 원본 배열/문자열 재순회 |

#### 자료구조 선택

- 존재 여부만 확인 → `Set`
- key-value 저장 필요 → `Map`
- 인덱스 접근 필요 → 배열 직접 순회

### 4단계: 개선 제안

- 현재 풀이의 장점 먼저 언급
- 문제가 있다면 **왜** 문제인지 설명 후 개선안 제시
- 대안 풀이가 있다면 복잡도 트레이드오프와 함께 제시

## 힌트 시스템

문제가 막힐 때 정답을 바로 요청하는 대신, **힌트를 요청**할 수 있습니다.

### 힌트 요청 방법

- `힌트 주세요` 또는 `hint` 라고 입력하면 단계적 힌트를 제공합니다.
- 힌트는 최대 3단계로 제공되며, 단계가 올라갈수록 구체적입니다.

### 힌트 단계

| 단계   | 내용                                                      |
| ------ | --------------------------------------------------------- |
| 힌트 1 | 어떤 자료구조 또는 알고리즘 접근법이 유효한지 방향만 제시 |
| 힌트 2 | 핵심 로직의 흐름(어떤 순서로 생각해야 하는지) 설명        |
| 힌트 3 | 구체적인 구현 아이디어 제시 (코드 X, 의사코드 수준)       |

### 힌트 원칙

- **정답 코드는 제공하지 않습니다** — 스스로 작성해야 진짜 실력이 됩니다.
- 힌트를 받은 후에는 직접 구현해보고, 풀이를 제출해 검토를 받으세요.
- 막혀 있는 부분이 어디인지 구체적으로 말해줄수록 더 유용한 힌트를 드릴 수 있습니다.
